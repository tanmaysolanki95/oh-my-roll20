# Fog of War Undo History Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add a server-persisted undo stack (max 50 entries) for fog of war paint operations, with a counter and Undo button in the DM sidebar.

**Architecture:** `fog_history: FogShape[][]` is stored as a new JSON column on `sessions`. Every fog paint pushes a snapshot of `fog_shapes` onto the stack before appending the new shape; undo pops the stack and restores the previous state. Both operations write `fog_shapes` and `fog_history` in a single DB update, which propagates to all clients via the existing `postgres_changes → setSession` realtime path — no new subscriptions needed.

**Tech Stack:** Next.js 15 (App Router), Supabase (Postgres + Realtime), Zustand, TypeScript. No test framework — verification is via `npm run build` and manual runtime testing.

### What to change

Replace `commitFogShape` with the version below. It reads `fog_history`, pushes the current `fog_shapes` snapshot (dropping the oldest if at 50), appends the new shape, then writes both columns in one DB update.

- [ ] **Step 1: Rewrite `commitFogShape` and add `undoFog`**

Replace the entire `commitFogShape` function and add `undoFog` directly after it:

```ts
const MAX_FOG_HISTORY = 50;

const commitFogShape = async (shape: FogShape) => {
  const s = useSessionStore.getState().session;
  if (!s) return;
  const prevShapes = s.fog_shapes ?? [];
  const prevHistory = s.fog_history ?? [];

  // Push current fog_shapes as a snapshot; drop oldest if at cap
  const trimmed = prevHistory.length >= MAX_FOG_HISTORY
    ? prevHistory.slice(1)
    : prevHistory;
  const newHistory = [...trimmed, prevShapes];
  const newShapes = [...prevShapes, shape];

  setSession({ ...s, fog_shapes: newShapes, fog_history: newHistory });
  const { createClient } = await import("@/lib/supabase/client");
  await createClient()
    .from("sessions")
    .update({ fog_shapes: newShapes, fog_history: newHistory })
    .eq("id", s.id);
};

const undoFog = async () => {
  const s = useSessionStore.getState().session;
  if (!s) return;
  const history = s.fog_history ?? [];
  if (history.length === 0) return;

  const newHistory = history.slice(0, -1);
  const previousShapes = history[history.length - 1];

  setSession({ ...s, fog_shapes: previousShapes, fog_history: newHistory });
  const { createClient } = await import("@/lib/supabase/client");
  await createClient()
    .from("sessions")
    .update({ fog_shapes: previousShapes, fog_history: newHistory })
    .eq("id", s.id);
};
```

Place `MAX_FOG_HISTORY` as a module-level constant above the hook function.

## Task 4: Wire `undoFog` in `SessionView.tsx` — counter, limit guard, Undo button

**Files:**
- Modify: `src/app/session/[id]/SessionView.tsx`

### Background: how `useFogPainting` is used today

`useFogPainting` is called inside `MapCanvas.tsx`, not `SessionView.tsx`. `SessionView.tsx` owns DM sidebar actions (`clearFog`, `toggleFog`, etc.) but doesn't call `useFogPainting` directly — that hook lives inside MapCanvas.

**The undo action needs to live in `SessionView.tsx`** (DM sidebar), but `undoFog` is returned by `useFogPainting` which is inside `MapCanvas`. The solution: implement `undoFog` as a standalone function directly in `SessionView.tsx` using the same pattern as `clearFog` — reading from `useSessionStore.getState()` and writing directly to the DB. This avoids prop-drilling through `MapCanvas` and keeps the DM action pattern consistent with existing code.

### What `clearFog` looks like today (model to follow):

```ts

  setSession({ ...current, fog_shapes: [] });
  await createClient().from("sessions").update({ fog_shapes: [] }).eq("id", sessionId);
};
```
